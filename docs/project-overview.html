WebCodex 项目详细说明

WebCodex 是一个 Web 版 Codex/Agent Demo：用户在浏览器发起代码任务，FastAPI 负责业务数据、文件索引、事件流和 Worker 调度，Python Worker 基于 OpenAI Agents Python SDK 运行 SandboxAgent，并在官方 Docker 沙箱中执行命令、编辑文件和查看图片。

仓库：Gitofxiongxiong/webcodex 后端：FastAPI + SQLite + Aliyun OSS Worker：Python + OpenAI Agents SDK 前端：Vite + React 19 + Tailwind 4

1. 项目定位

这个项目不是普通聊天页面，而是围绕“浏览器发起任务、后端持久化状态、Worker 执行 Agent、沙箱承载代码操作、前端实时展示过程”的完整 Demo。

当前版本已经覆盖用户注册登录、默认工作空间、工作空间文件管理、附件上传、对话历史、Run 创建与取消、SSE 事件重放、模型用量统计、Docker 沙箱命令执行，以及兼容 OpenAI 官方 Responses API 和本地 Codex Relay 两类模型接入路径。

业务元数据 SQLite 保存用户、工作空间、版本、消息、Run、事件、用量和账单流水。
文件内容 Aliyun OSS 按 SHA-256 内容寻址保存文件 blob，SQLite 只保存索引。
Agent 执行 Python Worker 调用 OpenAI Agents SDK，并将命令限制在 Docker /workspace。

2. 总体架构

Browser
  -> FastAPI Backend
      -> SQLite: users, workspaces, versions, conversations, runs, events, usage
      -> Aliyun OSS: content-addressed file blobs
      -> Python Worker process per run
          -> OpenAI Agents Python SDK SandboxAgent
          -> Official Docker sandbox mounted at /workspace
          -> Sandbox tools such as shell, apply_patch and view_image
      -> SSE event stream
  -> Browser UI renders assistant text, reasoning, tool calls, usage and exports

架构上有两个清晰边界：持久工作空间在后端和 OSS 中，Agent 的临时执行区在 Docker 沙箱中。Agent 需要先通过 workspace_tree / workspace_rg 发现文件，再用 workspace_import 将文件导入沙箱，编辑后如果启用了导出工具再显式写回工作空间。

4. 后端说明

4.1 FastAPI 主入口

backend/app/main.py 是后端核心，包含 CORS、健康检查、认证、工作空间、附件、对话、Run、SSE、Worker 内部接口、模型用量计费和 Codex Relay 路由。

用户与会话

支持注册、登录、登出和当前用户查询。密码使用 PBKDF2-SHA256 加盐哈希，登录后发放随机 session token，数据库只保存 token hash。

Run 调度

用户创建 Run 后，后端写入 run.queued 事件。启动时和后台循环会扫描 queued / starting 状态并拉起 Python Worker。

SSE 事件

Worker 将标准化事件 POST 到后端，后端写入 run_events。浏览器通过 /api/runs/{run_id}/events 消费事件，并可用 after 参数重放。

用量与计费

后端记录 model.usage 和上下文估算，按模型价格表计算 micro USD 和 micro credits，并写入 usage events 与 billing ledger。

4.2 配置层

backend/app/config.py 使用 pydantic-settings 从 ../.env 读取配置。它统一解析 SQLite 路径、Python Worker 入口、沙箱目录、Run 目录、Codex Relay auth 文件和 OSS 配置。

4.3 OSS 存储

AliyunObjectStore 是内容寻址 blob store。写入时计算 SHA-256，并保存到 {OSS_KEY_PREFIX}/objects/{sha前两位}/{sha}。如果同一个 SHA 已存在，则复用对象，减少重复存储。

4.4 Codex Relay

当没有官方 OpenAI API Key，且配置了 CODEX_RELAY_AUTH_FILE 时，后端可暴露 /codex-relay/v1/responses。该路由读取 docs/auth.json，必要时刷新 ChatGPT access token，再转发到 chatgpt.com/backend-api/codex/responses。

5. Worker 与沙箱

worker-py/webcodex_worker/main.py 是每个 Run 的执行进程。后端通过环境变量注入 RUN_ID、CONVERSATION_ID、WORKSPACE_ID、模型参数、API 地址、Worker Token 和 Docker 运行参数。

能力 实现 说明
模型运行 openai-agents 的 SandboxAgent、Runner 和 DockerSandboxClient 默认模型为 gpt-5.5，通过 Responses API 运行官方 sandbox agent。
工具调用 Capabilities.default() 暴露官方 sandbox 能力，包括 shell、apply_patch、view_image 等工具。
沙箱 DockerSandboxClient Worker 将工作空间物化到 data/runs/{run_id}/workspace，容器内根路径是 /workspace，产物目录是 /workspace/outputs。
工作空间桥接 materialize_workspace、create_diff、commit_workspace_changes Worker 先通过后端内部 API 拉取工作空间文件，运行结束后扫描差异并写回 OSS/SQLite。
附件处理 materialize_attachments 附件会复制到 /workspace/attachments，用户输入会附带可访问的本地路径，图片可由 view_image 工具查看。

当前 Docker 网络默认是 bridge，即容器内命令可访问网络。生产环境如果承载敏感代码，应进一步收敛网络、磁盘、凭据和导出策略。

6. 前端说明

前端位于 frontend/src/main.jsx，使用 React 19 单文件主应用组织大部分状态和 UI。它通过 localStorage 保存 API Base URL 与登录态，通过 Fetch 调后端 API，通过 EventSource 接收 Run 事件。

登录注册 未登录时显示认证页，登录后加载当前用户、默认工作空间、历史会话和模型目录。
聊天运行 提交消息时创建 conversation/run，显示 assistant 文本、思考过程、工具调用和停止按钮。
附件上传 支持拖拽、粘贴和文件选择，每条消息最多 20 个附件，上传后同时进入工作空间和模型输入链路。
工作空间 支持文件列表、目录、上传、复制、移动和刷新；聊天页也可打开文件树抽屉。

Markdown 渲染使用 streamdown 相关依赖和本地 MarkdownRenderer.jsx，用于展示模型输出、代码块、数学公式和 CJK 内容。

7. 数据与文件模型

7.1 SQLite 表

表 用途
users、auth_sessions用户账号、密码哈希、登录 session。
workspaces、workspace_versions每个用户一个默认工作空间，文件变更形成版本。
workspace_files、workspace_folders、file_ops当前和历史版本中的文件/目录索引，以及写入、复制、移动等操作历史。
conversations、messages、conversation_agent_items会话、用户/助手消息，以及 Agents SDK session replay item。
runs、run_events一次 Agent 执行及其可重放事件流。
attachments、message_attachments、run_attachments上传附件、消息关联和 Run 内部处理状态。
run_context_snapshots、model_usage_events、billing_ledger上下文估算、模型 token 用量、费用和金币流水。

7.2 文件内容与版本

每次写工作空间文件时，后端会把内容写入 OSS，然后创建一个新的 workspace version。新版本复制上一版本的文件快照，再替换被修改的路径，最后更新 workspaces.current_version_id。 

PUT /api/workspaces/{workspace_id}/files/{path}
  -> calculate sha256
  -> put OSS object: {OSS_KEY_PREFIX}/objects/{sha[0:2]}/{sha}
  -> create workspace version
  -> write workspace_files row
  -> append file_ops row

8. API 分组

分组 代表接口 说明
健康检查 GET /health 返回数据库、OSS、Worker、模型、Codex Relay 配置状态。
认证 POST /api/auth/register、POST /api/auth/login、GET /api/auth/me 浏览器用户入口，除 demo 兼容接口外，大部分用户 API 都依赖 Bearer token。
模型与费用 GET /api/models/catalog、GET /api/billing/usage、GET /api/runs/{run_id}/usage 查询模型目录、价格、单 Run 用量和用户累计用量。
工作空间 GET /api/workspaces、GET /api/workspaces/{id}/files、PUT /api/workspaces/{id}/files/{path} 文件列表、目录创建、上传、复制、移动、读取、下载和操作历史。
附件 POST /api/attachments、GET /api/attachments/{id}/content 聊天附件上传和下载，大小上限为 100 MB，单条消息最多 20 个。
对话与 Run POST /api/conversations、POST /api/conversations/{id}/runs、GET /api/runs/{id}/events 创建会话、发起 Agent 运行、取消运行、查询消息、查询事件历史和 SSE 实时流。
Worker 内部接口 /internal/workspaces/*、/internal/runs/*、/internal/attachments/* 只给 Python Worker 使用，依赖 WORKER_TOKEN，用于工作空间同步、事件写入、附件读取和 session item 保存。
Codex Relay POST /codex-relay/v1/responses 可选兼容路径，需要 CODEX_RELAY_ENABLED、本地 auth 文件和 relay token。

9.2 启动方式

仓库提供 PowerShell 快速启动脚本：

.\start.ps1
.\start.ps1 -Restart
.\start.ps1 -SkipInstall
.\start.ps1 -Open

脚本会创建 .env、Python 虚拟环境、安装后端/前端/Worker 依赖，并启动后端 http://127.0.0.1:8000 与前端 http://127.0.0.1:5173。

cd backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
pip install -e ..\worker-py
uvicorn app.main:app --host 127.0.0.1 --port 8000

cd ..\frontend
npm install
npm run dev -- --host 127.0.0.1 --port 5173

10. 关键流程

10.1 用户发起一次 Agent Run

1. 前端提交消息和附件 ID 到 POST /api/conversations/{conversation_id}/runs。
2. 后端校验用户、工作空间、附件归属，保存用户消息、Run 记录和 run.queued 事件。
3. 后端调度器将 Run 从队列拉起，启动 Python Worker 进程。
4. Worker 物化工作空间、读取历史 session item、处理附件、创建官方 Docker sandbox。
5. Worker 运行 SandboxAgent，模型可调用 shell、apply_patch、view_image 等官方 sandbox 工具。
6. Worker 将 SDK 事件归一化后 POST 到 /internal/runs/{run_id}/events。
7. 后端写入事件、用量、助手消息，并通过 SSE 推给前端实时展示。

10.2 附件处理流程

Browser upload
  -> FastAPI validates owner and size
  -> Aliyun OSS stores bytes by sha256
  -> SQLite attachments and workspace index are updated
  -> Worker downloads attachment bytes
  -> Worker writes file into Docker /sandbox
  -> Worker sends image/file model parts when provider supports it

10.3 沙箱导出流程

Worker 指令要求模型如果要交付生成文件，应写入 /sandbox/outputs/，并在最终回答中用 ![描述](sandbox://outputs/file.ext) 形式引用。后端在 assistant.message.done 时扫描这些引用，将沙箱文件导出到工作空间并产生 sandbox.exports.created 事件。

11. 限制与演进

当前限制

  • SQLite 适合 Demo 和单机验证，不适合作为高并发生产数据库。
  • SSE 事件使用 SQLite 轮询，生产环境应替换为 Redis Stream、NATS 或 Kafka。
  • OSS 必须配置后，工作空间文件和附件能力才完整可用。
  • 默认 Docker 网络允许公网访问，生产环境需要按任务、租户或安全等级细化网络策略。
  • 默认支持模型集合目前只包含 gpt-5.5，新增模型需要同步更新后端 catalog 和前端选项。
  • workspace_export 工具由 WORKER_ENABLE_WORKSPACE_EXPORT=true 控制，不是默认暴露。

推荐演进方向

  • 将 SQLite 替换为 Postgres，Run 事件流替换为专业消息队列或日志流系统。
  • 增强租户隔离：独立 OSS 前缀、短期凭证、细粒度 RBAC、Worker Token 轮换。
  • 完善工作空间版本能力：删除、回滚、diff、提交、分支和大文件分片上传。
  • 强化沙箱策略：禁网默认、allowlist、磁盘配额、运行时审计和高危命令审批。
  • 扩展模型目录，让后端成为唯一模型能力和价格来源，前端不再硬编码模型列表。
